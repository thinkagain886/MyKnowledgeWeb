🤖 LLM 基础原理
第一性原理：LLM 解决什么问题？原理是什么？
❓
解决什么问题？

传统程序只能处理规则明确的结构化任务。LLM 解决了「用自然语言理解和生成人类意图」的问题——让机器能理解模糊的、上下文依赖的、创造性的指令，并给出流畅的回应。

问题
自然语言理解与生成
核心能力
上下文推理 + 知识检索 + 格式生成
本质
下一个 Token 概率预测器
⚙️
核心概念速查
概念含义实际影响
TokenLLM 处理的最小单位，约 0.75 个中文字 / 1 英文单词影响成本和速度
上下文窗口模型一次能处理的最大 Token 数（如 128K）决定能传多少历史
Temperature控制随机性，0=确定性，1=创意，>1=混乱代码用 0，创作用 0.7
Top-P采样范围（nucleus sampling），配合 Temperature一般设 0.9-0.95
System Prompt角色设定和约束，在对话前注入定义模型行为边界
Few-Shot在 Prompt 中给示例引导输出格式格式稳定性大幅提升
Hallucination模型编造不存在的事实必须用 RAG 或 Grounding
💡
关键工程注意点
⚠️ 幻觉问题：模型不知道「不知道」，会编造答案。解法：RAG注入事实 + 要求模型引用来源 + 低Temperature
💡 上下文管理：超过窗口就会「遗忘」早期内容。解法：滑动窗口 + 摘要压缩 + 向量记忆
ℹ️ 成本控制：Token = 钱。System Prompt 越短越好，用 Cache 复用高频内容
✅ 确定性输出：生产环境 Temperature 设 0，需要 JSON 时在 Prompt 里强制约束格式
✍️ Prompt Engineering
第一性原理：Prompt 是与 LLM 沟通的编程语言
❓
解决什么问题？

同一个模型，用不同的 Prompt 能得到截然不同的结果。Prompt Engineering 解决「如何精确表达意图、约束输出、引导推理」的问题。本质是：把需求翻译成模型能最好理解的指令形式。

🔧 Tool Calling / Function Calling
第一性原理：让 LLM 具备执行能力，不只是说，还能做
❓
解决什么问题？

LLM 只能生成文字，无法查数据库、调 API、发邮件。Tool Calling 解决「LLM 的执行能力」问题——让模型决定「该调用哪个工具、传什么参数」，由代码层执行并把结果返回给模型，形成完整的推理-执行闭环。

Spring AI Tool Calling 实现
// 定义工具（Spring AI 用 @Tool 注解）
@Component
public class OrderTools {

    @Tool(description = "查询用户的套餐信息，需要用户手机号")
    public PackageInfo queryPackage(String phone) {
        // 调用真实业务接口
        return orderService.getPackageByPhone(phone);
    }

    @Tool(description = "查询流量使用情况")
    public TrafficInfo queryTraffic(String phone) {
        return trafficService.getUsage(phone);
    }
}

// 在 ChatClient 中注册工具
@Service
public class SmartAssistant {
    public String chat(String userMessage) {
        return chatClient.prompt()
            .user(userMessage)
            .tools(new OrderTools())  // 注册工具
            .call()
            .content();
    }
}
💡 工具描述（description）质量直接决定 LLM 调用准确率，要写清楚「什么时候用」「需要什么参数」
⚠️
注意事项
  • 参数校验：LLM 可能传错类型或缺参，工具层必须做防御性校验
  • 权限控制：工具不应该暴露危险操作（删除/转账），必须有权限检查
  • 幂等性：LLM 可能重复调用同一工具，写操作工具必须幂等
  • 超时处理：工具调用要有超时，避免 Agent 卡死等待
  • 错误返回：工具失败要返回明确错误信息，而不是抛异常
🧠 AI 记忆系统
第一性原理：让 AI 记住过去，才能有连贯的智能
❓
解决什么问题？

LLM 无状态，每次对话都从零开始。记忆系统解决「跨对话持续性」问题——让 Agent 记住用户偏好、历史行为、重要信息，像真正的助手而不是每次失忆的机器。

🗂️
四种记忆类型
短期记忆（Sensory/Buffer）
当前对话的上下文窗口内。滑动窗口保留最近 N 轮，超出就丢。实现：ChatMemory
长期记忆（Long-term）
持久化到外部存储（Redis/DB/向量库）。对话结束后提取关键信息写入，下次检索注入。
情节记忆（Episodic）
存储具体事件序列（"用户上次充值了100元"）。时间线索检索，还原事件上下文。
语义记忆（Semantic）
存储抽象知识和用户画像（"该用户偏好流量套餐"）。向量相似度检索。

Spring AI 记忆实现
// 短期记忆：保留最近10条消息
@Bean
public ChatMemory chatMemory() {
    return new InMemoryChatMemory();  // 或 RedisChatMemory
}

// 带记忆的对话（MessageChatMemoryAdvisor 自动管理）
public String chatWithMemory(String conversationId, String userMsg) {
    return chatClient.prompt()
        .advisors(new MessageChatMemoryAdvisor(
            chatMemory, conversationId, 10))  // 10轮窗口
        .user(userMsg)
        .call()
        .content();
}

// 长期记忆：对话结束后提取并存储
public void saveImportantInfo(String userId, String conversation) {
    // 让 LLM 从对话中提取关键信息
    String extracted = chatClient.prompt()
        .system("从以下对话提取用户关键信息，以JSON格式返回")
        .user(conversation)
        .call().content();
    // 存入向量数据库
    vectorStore.add(new Document(extracted,
        Map.of("userId", userId)));
}

🤖 Agent 是什么？
第一性原理：Agent = LLM + 工具 + 记忆 + 自主决策循环
❓
解决什么问题？

普通 LLM 是「问答机器」——给它一个问题，它给一个答案，仅此而已。Agent 解决「自主完成复杂任务」的问题——它能自己决定下一步做什么、调用什么工具、什么时候停止，像真正的工作者而不只是回答者。

类比：LLM 是一个博学的顾问（只能说），Agent 是一个能执行任务的助理（能说 + 能做）。

对比你的经验：支付系统中的「状态机 + 补偿机制」和 Agent 的「Loop + 工具调用」思路高度一致！

🔄 ReAct Agent
Reasoning + Acting — 最主流的 Agent 执行模式
❓
解决什么问题？

如何让 LLM 在执行过程中持续推理、动态决策下一步？ReAct（论文：2022 Google）解决「推理和行动交织」的问题——每步先 Thought（推理），再 Action（行动），再观察 Observation，形成螺旋上升的执行链。

📋
ReAct 执行示例
用户：帮我查一下手机号 13800138000 的套餐状态，如果快到期了提醒续费

Thought: 用户想查套餐状态，需要先调用查询工具
Action: query_package(phone="13800138000")
Observation: {"package":"100G月套餐","expire_date":"2025-01-25","remain_days":3}

Thought: 套餐还有3天到期，需要提醒用户续费。再查一下有哪些续费套餐
Action: list_packages()
Observation: [{"name":"100G月套餐","price":29},{"name":"200G月套餐","price":49}]

Thought: 已获取套餐信息，可以给出完整建议了
Final Answer: 您好！您的100G月套餐还有3天到期（1月25日）。
推荐续费方案：100G月套餐29元 或 升级200G月套餐49元。

// Spring AI 内置 ReAct Agent 支持
@Bean
public Agent reactAgent(ChatClient.Builder builder,
                         List<ToolCallback> tools) {
    return AgentBuilder.builder(builder)
        .tools(tools)
        .maxIterations(10)           // 最多10步，防死循环
        .build();
}

// 工具注册

public class PackageTools implements ToolCallback {

    @Tool(description = "查询用户套餐信息，参数为手机号")

        PackageInfo info = service.query(phone);
        return JsonUtil.toJson(info);
    }

    @Tool(description = "获取所有可购买套餐列表")
    public String listPackages() {
        return JsonUtil.toJson(service.listAll());
    }
}

⚠️
注意事项 & 避坑
  • 必须设最大迭代数：没有上限会死循环，推荐 5-15 步
  • 工具描述要精准：Thought 阶段 LLM 靠描述选工具，模糊描述导致调错
  • Observation 格式要简洁：返回结构化 JSON，不要返回大段文本
  • 失败处理：工具失败要返回错误信息而不是抛异常，让 LLM 能重试
  • 成本控制：每步都消耗 Token，复杂任务成本高，用缓存

1️⃣
Reflection 反思模式

问题：LLM 第一次输出质量不够好，如何自动优化？

原理：让 Agent 生成输出后，再用另一个（或同一个）LLM 来批评、指出问题，然后根据批评重新生成，循环直到满意。

// 生成-批评-改进循环
String code = generateCode(task);
for (int i = 0; i < 3; i++) {
    String critique = critiqueAgent.review(code,
        "找出这段代码的问题：安全漏洞、性能问题、代码规范");
    if (critique.contains("无问题")) break;
    code = improveAgent.fix(code, critique);
}
return code;
💡 代码生成、文章写作、SQL 优化等任务用反思模式，质量提升显著

Plan-and-Execute 规划执行模式

对比 ReAct：ReAct 是边想边做（每步决策），Plan-Execute 是先想全再做（全局规划）。

适合 Plan-Execute 的场景
  • 任务步骤相对固定
  • 需要并行执行多步骤
  • 希望用户能看到执行计划
适合 ReAct 的场景
  • 任务路径动态变化
  • 每步依赖上步结果
  • 探索性任务
// Plan-and-Execute 伪代码

// 可并行执行无依赖步骤

🔌 MCP — Model Context Protocol
AI 工具的「USB 接口」标准 — Anthropic 开源，行业正在采纳
❓
解决什么问题？

每个 AI 工具（Claude/GPT/Gemini）连接外部系统的方式都不同，形成割裂的孤岛。MCP 解决「AI 工具与外部系统的标准化互联」问题——定义统一的服务器/客户端协议，让任何 AI 工具都能连接任何外部服务。

类比：MCP 就像 USB 标准——有了 USB，任何设备都能连接任何电脑。有了 MCP，任何 AI 都能连接任何工具。

🏗️
MCP 架构
MCP Server（服务端）

暴露工具和资源的服务，你来写。把你的 Java 业务接口包装成 MCP Server，让 AI 能调用。

  • Tools：可调用的函数
  • Resources：可读取的数据
  • Prompts：可复用的提示词模板
MCP Client（客户端）

使用工具的 AI 应用。Claude Code、Cursor、Claude Desktop 都内置了 MCP Client。

  • 发现可用工具列表
  • 调用工具并获取结果
  • 读取资源数据

为什么自研 Spring AI 而不只用 Dify？
企业选型 · 面试必问
解决问题企业选型：低代码（Dify 拖拉拽，适合 PoC）vs 代码框架（Spring AI，RAG 逻辑内嵌 Java 服务）的权衡。
第一性原理RAG 不是独立 Demo，要嵌进现有 Java 微服务。Dify 适合验证，Spring AI 适合生产集成。
优化点

public void uploadAndIndex(MultipartFile file, String docType) {

        new InputStreamResource(file.getInputStream())).get();

        d.getMetadata().put("source", file.getOriginalFilename());
        d.getMetadata().put("docType", docType);  // FAQ / PACKAGE / NETWORK
        d.getMetadata().put("ossUrl", ossUrl);
    });
    // 4. 分片 + 向量化 + 存 Milvus（add 内部自动 embed）
    List<Document> chunks = faqSplitter.apply(docs);
    vectorStore.add(chunks);
    // 5. MySQL 持久化 chunkId 列表（删文档时用）

        .ossUrl(ossUrl).docType(docType).chunkIds(String.join(",", chunkIds)).build());
}

★ 分片策略（Chunking）
全称：Text Chunking / Document Splitting · 企业 RAG 最关键
⚠️ 踩坑：套餐说明若从中间切断，问「30 天加油包多少钱」可能搜不到完整价格，模型就编一个。代码写得好不如数据洗得好。
解决问题大模型单次吞吐上下文（Context）有限，整篇长文档直接喂进去会造成 Token 超限崩溃；且由于「大海捞针」效应，直接检索长文本会导致检索精度（Retrieval Accuracy）极低。
第一性原理大模型 RAG 的检索寻址单位是 Chunk。Chunk 语义不完整 = 喂给 LLM 的推理材料残缺 = 回答必然断章取义或编造。
核心分片策略矩阵（6 种）
1 · 对话式分块（FAQ Splitter）

解决问题：防止「问题」与「答案」被切断分家，导致大模型只看到答案却不知道在回答什么。

第一性原理：问与答（Q&A）是不可分割的最小业务逻辑单元。

咔咔快落地：《常见问题 FAQ.txt》，用特定分隔符（如 --- 或正则 Q:）扫描切分。

2 · Markdown 标题分块（MarkdownHeaderTextSplitter）

解决问题：防止套餐资费、核心条款、价格数字被拦腰截断，导致大模型拿到孤立的数字。

第一性原理：文档的结构化标题（# / ##）天然承载了其下方正文的宏观语义边界。

咔咔快落地：《套餐与加油包说明.md》，按标题切，利用「元数据下沉」机制，把 # 一级标题变成小 Chunk 的 Metadata 档案，不产生空块，保住资费前因后果。

3 · 递归字符分块（RecursiveCharacterSplitter）

解决问题：防止固定长度切分导致排查步骤（步骤 1/2/3）被机械切断，导致大模型给出的指南缺胳膊少腿。

第一性原理：文本的语义连贯性是层层退守的（段落 → 换行 → 句号 → 逗号 → 单字）。能用大刀切好，绝不用小刀剁碎。

咔咔快落地：《网络故障指南.txt》，代码优先找 \n\n（段落），长段落自动内部降级找 。 或 ， 切分，确保排查步骤锁在同一个 Chunk。

4 · 令牌文本切分（TokenTextSplitter）

解决问题：处理毫无格式、密密麻麻的长篇大论，防止因找不到换行符导致整个大块撑爆大模型输入限制（Token Overflow）。

第一性原理：大模型的吞吐极限是由 Token 数量这个物理铁律决定的。

特殊注意：属于盲目的「滑动窗口」，必须设置重叠区（Overlap）。Overlap 绝不判断语义，纯靠牺牲存储让前后块重复，给切刀边缘的信息买一份「双保险」——仅 TokenTextSplitter 盲切时才需要。

咔咔快落地：未分类长文章，default 兜底策略。

5 · 层级/父子分片器（Hierarchical / Parent-Child Splitter）新增

解决问题：彻底解决「高检索精度（要块小）」与「大模型推理前因后果（要块大）」之间的物理矛盾。

第一性原理：搜索（Search）的最小单位 ≠ 推理（Generation）的最小单位。短文本向量特征明显容易搜，长文本语境饱满大模型容易懂。

底层落地：子分片（~300 字）转成向量，塞入 Milvus，Metadata 带上 parent_id；父分片（~1500 字）纯文本不转向量，存在 Redis 或 MySQL 中。检索时「狸猫换太子」：用户提问 → 命中子分片 → 拦截子分片 → 拿着 parent_id 去 Redis 捞出父分片 → 喂给大模型。

咔咔快落地：《运营商合作跨境结算政策》《咔咔快法务合规合同》。

6 · 语义相似度切片器（SemanticSimilaritySplitter）新增

解决问题：解决作者思维跳跃、没有任何排版、自顾自写的「大杂烩」杂乱长文本。

第一性原理：文字之间的连续向量距离一旦发生骤降，即代表人类思维叙事发生了断裂，此处即为新切片的物理边界。

底层落地：本地调用 Embedding 向量模型（非大模型，成本极低）一句句算相似度。前一句聊「充值相关性 90%」不切，下一句突然跳跃到「吃兰州拉面相关性 10%」，咔嚓一刀切断。

咔咔快落地：无排版杂乱长文本。

public List<Document> splitByDocType(List<Document> docs, String docType) {
    return switch (docType) {
        // FAQ：令牌文本切分，控制体积极小，强力设置保护红线 5000 防止异常文本导致内存崩溃
        case "FAQ"     -> new TokenTextSplitter(400, 80, 5, 5000, true).apply(docs);
        // 套餐：Markdown 标题切分，元数据下沉，一级标题作为 Metadata 注入子块
        case "PACKAGE" -> new MarkdownHeaderTextSplitter(List.of("#", "##")).apply(docs);
        // 网络故障：递归字符切分，显式符号链「段落→换行→句号→逗号」层层退守
        // ⚠️ 若只传 "\n\n"，文章没换行将直接退化成硬切，丧失句号断句能力
        case "NETWORK" -> new RecursiveCharacterTextSplitter(
            List.of("\n\n", "\n", "。", "，", ""), 600, 50).apply(docs);
        // 兜底：未分类长文章，Token 硬切保命，确保大模型不爆 Token
        default        -> defaultTokenSplitter.apply(docs);
    };
}

方案适合场景优点缺点咔咔快建议

深挖：HyDE 和 RetrievalAugmentationAdvisor 是一回事么？
HyDE是一种思想（Hypothetical Document Embeddings）：用户提问太短/太口语化直接搜不到 → 先让 LLM 编一个「假设答案」，用假答案去搜，因为「答案找答案」比「问题找答案」语义更匹配。
RAAdvisor是 Spring AI 的模块化容器 Advisor，可以装各种 Transformer。
两者关系你可以在 RetrievalAugmentationAdvisor 里放一个 RewriteQueryTransformer 来实现类似 HyDE 的效果（确实是调模型改写，需结合业务引导词）。所以二者不是一回事，是「思想」与「实现容器」的关系。
要放进口诀么非常有必要！它属于检索阶段的「查询改写」步骤，是解决中文口语化提问召回率低的核心手段。
深挖：BM25 打分机制与 2026 混合检索（RRF）
BM25 适用场景向量搜语义强但搜不了精确词。咔咔快用户搜「设备号 ICCID 898601...」「错误码 E-404」「P10 套餐编号」时，向量模型把它当普通数字根本搜不准；BM25 擅长精确关键词匹配。
BM25 怎么打分基于 词频 TF（词在文档出现次数）× 逆文档频率 IDF（词在所有文档的稀有度）。一个词在少数文档频繁出现 → 得分高。并做文档长度归一化，长文档不因字多占便宜。
为什么用 RRF 不加权平均向量输出 cosine [-1,1]，BM25 输出无界正整数，直接加权 BM25 永远主导。RRF 只看排名位置，不看绝对分数，彻底绕开分数不可比。
RRF 公式 + k 值score(d) = Σ 1/(k + rank(d))。k=60 是默认基线（Elasticsearch/业界标准，无需调参，实测较纯向量 +14% 召回）；k=30~40 偏向 Top-1 精准度。两路都排前面的文档综合分最高。
Weaviate 陷阱v1.24 后默认融合算法从 RankedFusion 变为 RelativeScoreFusion，不显式指定会静默变更搜索质量。咔咔快用 Milvus 无此问题。

// 【咔咔快】SourceTraceAdvisor - 放在 QA Advisor 之后（order = MAX-1）
@Component
@Order(Ordered.LOWEST_PRECEDENCE - 1)
public class SourceTraceAdvisor implements CallAdvisor {
    @Override
    public ChatClientResponse adviseCall(ChatClientRequest req, CallAdvisorChain chain) {
        List<Document> docs = (List<Document>) req.context()
            .get(QuestionAnswerAdvisor.RETRIEVED_DOCUMENTS);  // 取检索到的 chunk
        String sources = docs.stream()
            .map(d -> d.getMetadata().get("source").toString())
            .distinct().collect(joining("、"));  // 去重
        ChatClientRequest enriched = req.mutate()
            .prompt(req.prompt().user(req.prompt().getUserMessage()
                .getText() + "\n[参考来源: " + sources + "]")).build();
        return chain.nextCall(enriched);
    }
}

Q1：介绍你的 RAG 项目（必问 · 背这段）

解决什么问题：咔咔快随身 WiFi 平台，用户每天高频咨询实名认证、充值失败、加油包怎么买、网络信号差，人工客服压力大。传统关键词搜索搜不到语义相关内容（「支付没成功」搜不到「充值失败处理」）。

怎么做的：Spring AI + 通义千问 + Milvus。后台运营上传 FAQ 原文件（PDF/Word）→ 阿里云 OSS → Tika 解析 → 按 docType 选分片策略（FAQ 问答对切、套餐 Markdown 标题切、长政策/合同类走父子分片「子检索父推理」）→ text-embedding-v3 向量化 → 存 Milvus。App 端多选知识库文档（metadata filterExpression）→ 检索 Top-3 → 注入 System Prompt → 通义千问流式回答 + 标注来源。FAQ 类走 RAG，查套餐余额/流量走 Tool Calling。

优化效果：覆盖 80%+ 高频咨询，夜间无人客服 99% 自动回复。通过分片策略优化 + threshold 降低（0.8→0.35），召回率从 60% 提升到 90%+。

Q2：RAG 和微调（Fine-tuning）怎么选？

RAG 适合数据频繁更新（咔咔快套餐每季度调价，换文档即可，成本极低）且能溯源；微调适合固定风格/领域，更新慢费用高且无法溯源。咔咔快当前只用 RAG，FAQ 迭代快无需微调。两者可结合：RAG 提供最新内容，微调统一输出风格。

Q3：向量库为什么选 Milvus？pgvector/Redis/ES 区别？

选型核心是数据量、基础设施、是否需混合检索。Milvus 专业向量库，Spring AI 原生支持，FLAT/HNSW 可切换。pgvector 适合向量<500万且已有 PG。Redis 极快但内存贵，做缓存层。ES 自带 BM25 适合混合搜精确词（ICCID）。咔咔快选 Milvus 因教程链路成熟，后续需混合检索加 ES 双写 RRF 融合即可。

Q4：怎么防止 RAG 幻觉？（2026 高频三段式）

两类幻觉：检索为空瞎编；召回错误 chunk 基于错误材料推理。客服场景零容忍。三层治理：① 检索层：分片语义完整 + Metadata 缩范围 + threshold=0.35 + 可选 Rerank；② 生成层：System Prompt 硬约束「无信息回拨 400」，Temperature=0；③ 兜底：allowEmptyContext=FORCE 检索为空直接返回不调大模型。效果：套餐价格幻觉从 10% 降到 0，XXL-Job 周跑 RelevancyEvaluator <90% 告警。

Q5：Top-K 不够 / 混合检索 / Rerank 具体怎么做？

① MVP：Metadata 多选 + threshold=0.35 + 合理分片；② 混合检索：Milvus 向量 Top10 + ES BM25 Top10，RRF（公式 1/(60+rank) 只看排名不看分数）融合取 Top5；③ Rerank：前两步仍不够时粗筛 topK=20 → DashScopeRerankAdvisor 精排留 3 条。不要上来就 Rerank，先把分片和阈值优化好。

Q6：Metadata 多选怎么实现？

上传时写 metadata.source=原文件名。App 多选传 List<String> sources，后端构建 FilterExpression = "source in ['实名.pdf','充值.docx']" 放入 SearchRequest。未选则不走 RAG。后端必须二次 Shiro 权限校验，防抓包传越权 source。

Q7：知识库文档怎么更新？三删一致性？

传错文档 → MySQL 查 chunkIds → Milvus 批量 delete → MySQL 删记录 → OSS 删物理文件，@Transactional(rollbackFor=Exception.class) 保证前三步一致，OSS 删失败异步补偿。套餐调价不直接改，传新版 v2025Q2 多版本共存，App 多选对比，Prompt 引导「以最新版为准」。chunkIds 上传时就存 MySQL。

Q8：RAG 准确率怎么评估？

不靠感觉。① RelevancyEvaluator 判断问题-上下文相关性；② FactualityEvaluator 判断幻觉；③ 人工标注 50~100 条「问题→正确 chunkId」算召回率；④ RAGAS 自动评 faithfulness。XXL-Job 周跑 <90% 钉钉告警。低分逐环节反推：先查分片 → 再查阈值 → 再看 Prompt。90% 问题在数据。

Q9：Spring AI 的 Advisor 执行顺序？

Advisor 本质 AOP 拦截器，通过 @Order 控制。SourceTraceAdvisor 必须在 QuestionAnswerAdvisor 之后执行（LOWEST_PRECEDENCE - 1），因为要等它把 chunk 检索出来放到 context 里，才能读 metadata.source 拼到 Prompt。

Q10：RAG 和 Tool Calling 怎么配合？意图路由设计？

不是所有问题适合 RAG。「怎么实名」→ RAG；「我的流量还剩多少」→ 查实时数据库。意图分类器判断 FAQ 类（怎么操作、什么是）→ RAG；业务查询类（我的、查、余额）→ Tool Calling 调微服务查 MySQL。两路可同一次对话混用。咔咔快 MVP 用关键词规则（含「我的/查/余额」→ Tool），其余走 RAG。

Q11：分片策略怎么选？（核心高频）

Chunk 是检索单位，语义不完整 = 材料残缺 = 回答必然错。没有万能策略，按文档类型选：FAQ → 对话式一问一块；套餐/技术文档 → Markdown 标题切；故障指南 → 递归字符分块（段落→句号层层退守）；无格式长文 → TokenTextSplitter 兜底；长政策/合同 → 父子分片（子分片 ~300 字入 Milvus 检索，父分片 ~1500 字存 Redis 送 LLM，解决「搜用小、推理用大」矛盾）；杂乱无排版 → 语义相似度切片（Embedding 算相邻句相似度，骤降处切断）。

原则：宁可多切几个小 chunk，不要两主题挤一块。上线召回率低第一步查分片，比调参有效 10 倍。2026 注意：Markdown/FAQ 语义切分已做好则不必盲目堆 overlap，仅 TokenTextSplitter 盲切才需要 Overlap 兜底。企业级应提供分片预览 + 运营手动合并/编辑后增量向量化。

Q12：大模型参数 Temperature/TopK/MaxTokens 怎么设？

客服场景：Temperature=0（必须准确不发散）；TopP=0.9（默认，T 为主控）；MaxTokens=500（约 300 字省费）；similarityThreshold=0.35（中文别设 0.8）；topK=3~5（FAQ 够用，Rerank 粗筛设 20）。调参前先把分片和召回优化好，不要用参数掩盖数据质量问题。

@Service
public class CustomerServiceAgent {

    @Autowired ChatClient chatClient;
    @Autowired VectorStore vectorStore;
    @Autowired ChatMemory chatMemory;
    @Autowired OrderQueryTools orderTools;

    public String chat(String userId, String message) {
        return chatClient.prompt()
            .system("""
                你是咔咔快随身WiFi的智能客服。
                只回答WiFi相关问题，无法解决时引导拨打客服电话。
                回答简洁，控制在100字以内。
                """)
            .advisors(
                // RAG：自动检索相关知识库
                new QuestionAnswerAdvisor(vectorStore,
                    SearchRequest.defaults().withTopK(3)),
                // 记忆：保留最近10轮对话

                    chatMemory, userId, 10)
            )
            .tools(orderTools)   // 工具：可查询套餐/流量
            .user(message)
            .call()
            .content();
    }

Q1：说说你对 RAG 的理解，你们是怎么实现的？

答：RAG 解决 LLM 不了解私有数据的问题——不微调，推理时检索外部知识注入 Prompt。分两阶段：Indexing（解析→分片→Embedding→Milvus）和 Retrieval+Generation（Metadata 过滤→Top-K 检索→拼 Prompt→LLM）。

我在项目中用 Spring AI 实现了完整企业级 RAG：Tika 多格式解析、按业务选分片策略、Metadata 知识库隔离、幻觉 Prompt 约束、来源追溯 Advisor。详见本指南「企业级 RAG」章节。

Q1.1：企业级 RAG 和 Demo 有什么区别？

答：Demo 只用默认 TokenTextSplitter + 全局 topK=5；生产级需要：多策略分片+预览、Metadata 隔离、跨 chunk 聚合（Text-to-SQL/分而治之）、幻觉治理、RBAC 权限、版本化更新、量化评估+告警。核心认知：代码写得好不如数据洗得好，80% 精度靠分片和 Embedding。

Q2：ReAct Agent 的原理是什么？

答：ReAct = Reasoning + Acting，交替进行推理和行动。每步先输出 Thought（分析当前情况，决定下一步），然后 Action（调用工具），获取 Observation（工具返回结果），再基于结果继续 Thought，循环直到得出 Final Answer。

核心价值是「动态决策」——不需要预先规划所有步骤，每步根据最新观察结果灵活调整路径。必须设置最大迭代步数防止死循环。

Q3：Spring AI 和 LangChain4j 怎么选？

答：两者都是 Java AI 框架，选择取决于场景。国内阿里云体系首选 Spring AI Alibaba，与通义千问和 Spring Cloud 生态完美融合，团队学习成本最低。

如果需要复杂 Agent 链或者需要切换多家模型提供商，LangChain4j 的 AiServices 声明式接口和 60+ 模型支持更有优势。实际项目中可以组合使用，Spring AI 做微服务集成，LangChain4j 做 Agent 逻辑。

Q4：如何防止 LLM 幻觉？RAG 场景怎么答？

答：RAG 幻觉分两类——检索为空：Prompt 约束「查不到就说不知道」+ allowEmptyContext=false；检索错误：Garbage In Garbage Out，从分片→Embedding→Metadata 过滤→Rerank 全链路优化。通用层面：Temperature=0、要求引用来源、关键数字后处理验证。

Q4.1：RAG 分片策略怎么选？

答：没有万能策略。FAQ 按问答对切；技术文档/套餐按 Markdown 标题；故障指南用递归字符分块（显式符号链段落→句号→逗号）；无格式长文用 TokenTextSplitter 兜底；长政策/合同用父子 chunk（子检索父送 LLM，解决搜用小推理用大矛盾）；杂乱无排版文档用语义相似度切片（Embedding 算句间相似度骤降处切断）。禁止纯固定长度切分。2026 注意：语义切分已做好不必盲目堆 overlap，仅 Token 盲切才需要。企业级应提供多策略 + 分片动态预览（高亮「元/GB」资费元素）+ 运营手动干预后再增量向量化。

Q4.2：知识库文档怎么更新？

答：传错文件→三表一致删除（DB + vectorStore.delete + OSS）加事务；文档改版→保留旧版，上传新版本（文件名带 v1/v2），对话多选+来源追溯，支持新旧对比。更多 RAG 面试题见「企业级 RAG」章节底部。

Q5：你是如何用 AI 提升开发效率的？

答：我遵循「需求→PRD→验收标准→技术方案→Plan/编码→Review→测试」的 AI 驱动流程。用 Cursor 配合 .cursorrules 约束 AI 生成符合项目规范的代码，大幅减少 Review 成本。复杂功能用 Composer 多文件生成，调试时把错误日志和代码一起贴给 AI 快速定位问题。

整体来说，重复性代码（CRUD、DTO 转换、测试用例）效率提升 3-5 倍，主要精力集中在业务逻辑设计和架构决策上。
